Trident API Overview

Stream是Trident中的核心数据模型.

  • Stream，别名TridentTuple。
  • 一个stream被划分成很多partition：partition是stream的一个子集，里面可能有多个batch，一个batch也可能位于不同的partition上

Trident 有5种操作：

  • Partition-local operations：对每个partition的局部操作，不产生网络传输
  • Repartitioning operations：对数据流的重新划分（不改变内容），产生网络传输
  • Aggregation operations： that do network transfer as part of the operation
  • Operations on grouped streams：作用在分组流上的操作
  • Merges and joins

Partition-local operations

对每个partition的局部操作包括：function、filter、partitionAggregate、stateQuery、partitionPersist、project等。

Aggregator

最通用的聚合接口是 Aggregator，如下

  • init method is called before processing the batch. The return value of init is an Object that will represent the state of the aggregation and will be passed into the aggregate and complete methods.
  • aggregate method is called for each input tuple in the batch partition. This method can update the state and optionally emit tuples.
  • complete method is called when all tuples for the batch partition have been processed by aggregate.

Repartitioning operations

Repartition操作可以改变tuple在各个task之上的划分。Repartition也可以改变Partition的数量。Repartition需要网络传输。

  1. shuffle：随机将tuple均匀地分发到目标partition里。
  2. broadcast：每个tuple被复制到所有的目标partition里，在DRPC中有用 — 你可以在每个partition上使用stateQuery。
  3. partitionBy：对每个tuple选择partition的方法是：(该tuple指定字段的hash值) mod (目标partition的个数)，该方法确保指定字段相同的tuple能够被发送到同一个partition。—相当于FieldGrouping方式
  4. global：所有的tuple都被发送到同一个partition。
  5. batchGlobal：确保同一个batch中的tuple被发送到相同的partition中。
  6. patition：该方法接受一个自定义分区的function（实现backtype.storm.grouping.CustomStreamGrouping）

Aggregation operations

2种方式：Trident中有aggregate()和persistentAggregate()方法对流进行聚合操作。

  • aggregate()在每个batch上独立的执行
  • persistemAggregate() 对所有batch中的所有tuple进行聚合，并将结果存入state源中。

性能调优的点
同在partitionAggregate中一样，aggregate中的聚合器也可以使用链式用法。但是，如果你将一个CombinerAggregator链到一个非CombinerAggregator后面，Trident就不能做局部聚合优化。
这句话是说在.chainedAgg() 和 .chainEnd()之间，如果有多个聚合器，若他们不全是实现的CombinerAggregator，那么这个东西不能够被优化。
关于persistentAggregate的用法请参见Trident state一文。

思考：

  • When a join happens between streams originating from different spouts, those spouts will be synchronized with how they emit batches. That is, a batch of processing will include tuples from each spout.
  • how do you do something like a “windowed join”
  • 为了解决上面的问题， you would make use of partitionPersist and stateQuery. The last hour of tuples from one side of the join would be stored and rotated in a source of state, keyed by the join field. Then the stateQuery would do lookups by the join field to perform the join.
